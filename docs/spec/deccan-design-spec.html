01Premise

A document standard exists to settle questions in advance, so that the next forty Deccan documents do not relitigate the same decisions about font, colour, page furniture, and tone. Version 2.0 settles those questions on the OS-native font stack, the swiss_design_at_deccan colour tokens, and an editorial register.

Version 1.0, the Aptos-first plan, was correct in principle: standardise on a single, well-engineered face that resolves on every endpoint. In practice, the Aptos plan introduced two costs that did not pay back. It made every Deccan document depend on a Microsoft 365 license, by way of the Office Font Service. It required an MSI deployment to install Aptos system-wide on machines whose Office version did not auto-bundle the face, and an Intune compliance lane to confirm the install. Neither cost is large. Both compound across every contractor laptop, partner workstation, and external reviewer that touches a Deccan document.

Version 2.0 takes a different position. The primary face is whatever the operating system provides. On Windows 11 — the bulk of the Deccan fleet — that resolves to Segoe UI Variable, a well-engineered variable font with optical sizing that ships with the OS and renders cleanly from 8 points to 64. On macOS — the M-series Studios in plant labs and the MacBooks in field roles — system-ui resolves to San Francisco. On any browser or email client on either platform, the chain falls back through documented stops without requiring an install. No MSI, no font binary in the artifact, no M365 dependency.

The aesthetic this produces is not a compromise. Segoe UI Variable was designed for screen reading at corporate-monitor density, and it carries the same swiss discipline the original swiss_design_at_deccan system was built around — clean, neutral, geometric, optically corrected. The non-deccan-design endpoint variation in the v0.1 brief specimen, which fell through to Segoe UI by accident of font absence, demonstrated this. Version 2.0 makes that fallback the deliberate primary.

03Type system

Two faces, both resolved through OS-native chains, both selected for their behaviour at the densities that matter — corporate-fleet 1080p and 1440p screens, and 300+ DPI print.

Sans

Body, heading, and UI text resolve through the chain below. The first installed family wins.

font-family:
  'Segoe UI Variable Display',
  'Segoe UI Variable Text',
  'Segoe UI Variable',
  'Segoe UI',
  system-ui,
  -apple-system,
  BlinkMacSystemFont,
  sans-serif;

On Windows 11 the chain resolves to Segoe UI Variable, a variable font with an optical-sizing axis. With font-optical-sizing: auto set on body, the rendering engine selects the Display variant for headings, the Text variant for body copy, and the Small variant for legal-sized labels. The result is type that holds its rhythm across the full scale of a document without manual intervention. Earlier Windows versions resolve to the static Segoe UI faces, which are visually compatible.

On macOS the chain resolves to San Francisco via system-ui, with -apple-system and BlinkMacSystemFont as Webkit and Chromium-on-Mac belt-and-braces. San Francisco carries its own optical-sizing behaviour through SF Pro Text and SF Pro Display.

The terminal fallback is the generic sans-serif, which any browser will resolve. The chain contains no Helvetica, no Arial, no Calibri, no Verdana — the generic fallback is preferred over a worse-than-default named family.

Mono

Code blocks, code chips, mono labels, and tabular numerals resolve through:

font-family:
  'Cascadia Mono',
  'Cascadia Code',
  Consolas,
  'SF Mono',
  Menlo,
  'DejaVu Sans Mono',
  'Liberation Mono',
  ui-monospace,
  monospace;

On Windows 11, Cascadia Mono ships with the Windows Terminal and is the natural pair for Segoe UI Variable. Cascadia Code is the same face with programming ligatures enabled — appropriate for code-heavy documents, but Mono is the document default to keep ligatures from inserting themselves into prose. Consolas is the Windows 7/8/10 fallback. On macOS the chain resolves to SF Mono via the named entry. The terminal fallback is ui-monospace, then the generic.

Weight and the size–weight relationship

Variable-font systems make weight a continuous axis. The convention here is to lighten weight as size grows, so that large type does not overwhelm the page.

  • Body and lists. 400 (regular).
  • Lead paragraph. 400 (regular), held at --stone-700 for tonal contrast against body.
  • H4. 600 (semibold) — small-format heading needs weight to register as heading.
  • H3. 600 (semibold).
  • H2. 500 (medium) — at 32px, semibold becomes too dense.
  • H1. 350 (light-medium) — at 44px, anything heavier shouts.
  • Display (cover, end page). 350 (light-medium).
  • Inline emphasis. 600 for <strong>; italic for <em>.
  • Mono labels. 600 — at 11px, semibold is the only weight legible on 1080p Windows.

Measure

Body prose is held to a 70ch measure on screen — roughly 70 characters per line, the editorial sweet spot at which the reader's eye does not lose its place tracking from line end to line beginning. Print follows the live content area, which on Letter at 0.8" margins gives roughly the same character count at the print body size.

Tables, callouts, code blocks, and pull quotes all inherit the same measure on screen and break to full content width in print, where margins already provide the constraint.

04Colour tokens

One accent. One reserved colour. Ten stones. The discipline is in what is left out.

Deccan Blue

The single accent. Used for: primary headings on the cover and end page; left-rule on callouts and pull quotes; underline on links; focus rings; section accent rules; the wordmark mark; the table-header underline. Used at four opacity stops so that one hue carries the full weight of hierarchy without introducing a second hue.

Deccan Green — confined use only

The Deccan Green #71BF4D is the secondary brand colour. It appears only in two contexts: as a constituent of the Deccan logo where the official mark is reproduced, and as a colour token in content explicitly about sustainability — environmental reporting, BRSR submissions, ESG disclosures, plant emissions narratives. Outside those contexts, it does not appear. It is not a UI accent. It is not a button colour. It is not a "secondary brand colour" available to designers. It is a flag for one thing.

Discipline rule

Documents that introduce #71BF4D outside the logo or sustainability content fail review. The reviewer's question: "Where in this document does this colour signal sustainability?" If the answer is "nowhere," remove it.

Paper

--paper resolves to #FFFFFF. It is the cover background, the end-page background, the table cell background, and the print page background. It is not aliased to a stone. White is white.

05Spatial system

Grid

The screen layout is a 12-column grid with a 24-pixel gutter, contained at 1180px maximum and side-padded with clamp(20px, 4vw, 64px). The grid is implicit — content uses CSS Grid where structure benefits from it (cover meta, comparison tables, masthead-style blocks) and prose flows in a single column at the body measure of 70ch.

Spacing scale

Spacing follows an 8-pixel base unit with a Fibonacci-ish progression for the larger steps. Components use the tokens directly so that revisions adjust globally.

Margins, print

Letter and A4 paper, 0.8-inch outside margins, 1-inch bottom margin to clear the running footer. The live content area on Letter is 6.9" × 9.2", which delivers the editorial measure at the print body size without further constraint. Long tables, code blocks, and figures break to full content width.

06Components

The component set is small. Each component has one reason to exist and one rule about when to use it.

Cover page

Three regions, top to bottom: a wordmark (3-pixel Deccan Blue rule, 56px wide, paired with the company name in semibold); a centred title block (eyebrow chip on stone-100, large light-weight title in Deccan Blue, subtitle at stone-700); a five-column metadata strip across the bottom (document type, prepared by, date, version, classification — each label in mono uppercase at --fs-tiny, each value in semibold at --fs-small).

The cover is mandatory. It carries no header, no footer, no page number. The first body section starts on the page after.

End page

Self-contained: centred wordmark, centred Deccan Blue rule, centred contact line, optional classification reminder in mono caps. No header, no footer, no page number. Exists to bookend the document and signal "this is where it ends."

Heading hierarchy

Four levels. Each level has a defined typographic identity. H1 carries an eyebrow with the section number (decimal-leading-zero), and a 2-pixel solid bottom rule in stone-900 — the most assertive structural element in the document, used sparingly.

An H3 looks like this

H3 is semibold at --fs-h3, intended for the second level of structural division within a section. H4 is also semibold but at --fs-h4, intended for inline emphasis within a long H3 block — the level at which a sub-topic earns a heading rather than a topic sentence.

Lead paragraph

A lead paragraph opens a section at --fs-lead, weight 400, colour --stone-700, line-height 1.5, measure 60 characters. It states the claim of the section in one sentence (or two, if necessary) before the body argues it.

The lead is optional. Use it when a section needs a thesis statement; skip it when the H1 plus the first body paragraph carry the section without one.

Callouts

Stone-100 fill, three-pixel Deccan Blue left rule, uppercase mono label in stone-500, body in body type. Used for: rules ("Provisioning rule"); warnings that need to register without shouting ("Non-negotiable"); editorial asides that interrupt the main argument deliberately. The variant .callout.muted uses a stone-700 left rule for asides that should not draw the eye to a brand colour — discipline notes about the system itself, for instance.

Code chip and code block

Inline code chip — like this — is mono on stone-100 with light horizontal padding. A code block is the same colour pair at scale, full measure, no border. The discipline rule: mono and stone always travel together. Mono on a white background reads as a typo; stone-100 on sans body reads as a callout. Together they read as code, which is the only thing they should ever read as.

Pull quote

Used to break editorial rhythm in a long section. Indented with a Deccan Blue left rule (3 pixels), held at 60 characters, set at 22 pixels in stone-700. Carries no quote marks — the rule and the type treatment do the work.

Tables

Headers in mono uppercase at 11 pixels, on stone-100 fill, with a 2-pixel solid stone-900 underline. Body cells in body type, banded by alternating between paper and stone-50, with 1-pixel stone-200 row separators. Borders are kept off the columns; the eye uses the band and the separator to track. Tables hold their measure on screen and expand to full content width in print.

Lists

Bulleted lists use the default disc marker tinted to stone-500. Numbered lists use decimal markers in the same tint. Definition lists pair a semibold term with a stone-200-bordered description block — the most legible treatment for short glossaries and rule sets.

Links

Deccan Blue text with a 1-pixel underline at --deccan-blue-30, offset 3 pixels below the baseline. On hover, the underline strengthens to full Deccan Blue. On focus, a 2-pixel --deccan-blue-30 outline at 2-pixel offset — visible, accessible, brand-aligned.

07Print rules

Print is the high-fidelity end of the medium and the medium with the strictest rules. The system carries a single @page stylesheet that handles every print case: PDF export from a browser, PDF print from Word, and direct browser print to a hardware printer. Documents do not need per-document print stylesheets; if one is needed, the system needs the rule, not the document.

Page format

Letter paper, portrait, 0.8-inch outside margins, 1-inch bottom margin. A4 follows the same proportions and renders without alteration on European-paper printers because the live content area scales with the paper.

Running footer

Mono, 8.5pt, stone-500. Left: Deccan Fine Chemicals · Confidential. Right: bare-integer page number. Bare integer, not "Page 12" and not "Page 12 of 47." The integer is the convention; explanatory wrappers add noise. The footer is suppressed on the cover (@page :first) and on the end page (which is in its own document section with no footer).

Page breaks

Every H1 forces a page break before, by default. Cover page-breaks after itself. End page page-breaks before itself. Within a section, the rendering engine uses standard widow/orphan handling with page-break-inside: avoid on callouts, code blocks, pull quotes, tables, and TOC entries — these are the elements that look broken when split across pages, and the cost of forcing them onto a single page is acceptable.

Print background

Body background is #FFFFFF in print, regardless of the screen background. Stone-50 is a screen-only convenience that softens the page in a browser; it is not a print convention. Stone tints in print appear only inside callouts, code blocks, banded table rows, and chips — the contexts where the tint is the point.

Print body size

Body type drops from 17px on screen to 10.5pt in print. The reduction is not arithmetic — print pixels-to-points conversion does not work cleanly because the rendering engine uses a different metric for print. The 10.5pt body is the size that produces 70-character lines on Letter at 0.8-inch margins, which is the editorial target.

08Screen rules

The corporate fleet runs primarily on 1080p (1920×1080) and 1440p (2560×1440) external monitors, with a smaller fraction of laptop-internal panels at 1080p or 1200p. The system optimises for these densities, not for retina-class displays where almost any treatment looks good.

Body size on screen

17 pixels at the root size of 16. The system does not rely on user font-size preferences scaling the document — it sets --fs-body at 17px directly. The reasoning: a 16px body, the typical web default, sits at the lower bound of comfortable reading on 1080p Windows where Segoe UI Variable is the rendered face. 17px lifts it into the editorial-comfortable range without overshooting the 18px point at which mono code chips begin to look oversized in line with the body.

Optical sizing

The body has font-optical-sizing: auto set globally. This activates Segoe UI Variable's optical-sizing axis on Windows and SF Pro's equivalent on macOS. The effect is mostly invisible — letterforms thicken slightly at small sizes and thin slightly at large sizes — but cumulative across a document, it produces the rhythm that distinguishes editorial from corporate-deck typography.

Subpixel rendering

The body has -webkit-font-smoothing: antialiased and -moz-osx-font-smoothing: grayscale set. On macOS this disables sub-pixel positioning, which makes type slightly thinner but visually consistent across regions of the screen. On Windows (Chromium and Edge), antialiased rendering is the default already; the rule is harmless. Removing the rule causes macOS Safari/Chromium to render type heavier, which fights the editorial weight discipline.

Hover and focus

Links carry a Deccan Blue underline at 30% opacity by default; hover deepens to full Deccan Blue. Focus produces a 2-pixel outline at 30% opacity with 2-pixel offset — visible against any document background, accessible to keyboard navigation, brand-aligned. Hover styles do not exist in print and do not exist on touch interfaces; the focus state covers the accessibility case.

Reduced motion

The system has no animation. prefers-reduced-motion is therefore moot for this version. Future versions that introduce motion components must respect it.

09Tone and voice

Documents issued under this system are corporate communications — internal standards, technical guides, audit-grade specifications, policy briefs. They are read by engineers, auditors, customers, and regulators. The register matches.

Required register

Direct, declarative, third-person. State the position, then the rationale. Active voice for actions; passive only when the actor is irrelevant or an established corporate process. Specific over generic — version numbers, dates, file paths, policy clauses, named systems.

Banned patterns

The patterns below are not acceptable. They appear in machine-generated draft text and in junior business writing; they do not appear in Deccan documents.

  • Conversational subtitles in headings (": in plain English", ": the EOL clock is the deciding factor"). Use plain noun-phrase headings.
  • First-person narrator asides ("Let me walk you through…", "I want to show you…"). State the content directly.
  • Filler openings ("Here's the thing:", "It's worth noting that…", "Let's start by…"). Start with the substantive sentence.
  • Casual transitions ("That said,", "With that out of the way,", "Now, let's…"). Use a section break or a topic-sentence transition.
  • Decorative emoji in body content. Use words: "Required", "Prohibited", "Note".
  • Exclamation marks in body prose. Use a period.
  • "Deep dive", "let's dive in", "rabbit hole", "TL;DR". Use plain language.
  • Mental-exercise framings ("A useful way to think about this is…", "Imagine if…", "Think of X as Y"). State the idea. If a comparison is genuinely needed, label it: "Analogy:".
  • Self-reference to the document ("This guide will teach you…", "By the end of this section you'll know…"). The document's existence implies its purpose.
  • Em-dash narrator interjections ("— and here's the kicker —"). State the consequence directly.

The read-aloud test

If a sentence would feel out of place in a board paper, an audit response, or a regulator submission, rewrite it. The corporate-document register is unambiguous; the test is fast.

10Banned and prohibited

Banned faces

The following sans-serif faces do not appear in any Deccan document, regardless of context:

  • Helvetica and Helvetica Neue — off-discipline; visually heavy; licensing complexity.
  • Univers — same.
  • Arial — Microsoft's Helvetica clone; the textbook tired-default face.
  • Calibri — Office 2007–2023 default; replaced upstream by Aptos; legacy connotation.
  • Verdana — designed for low-resolution CRT screens; over-wide, over-spaced; out of place on modern displays.
  • Times, Times New Roman, Garamond, Georgia — serif faces; the system is a sans-serif system.

The CSS chain in §03 explicitly excludes all of these. Documents that introduce them by overriding the chain fail review.

Banned monospace faces

  • Courier and Courier New — visually heavy; typewriter connotation; out of place in modern technical writing.
  • Lucida Console — legacy Windows; replaced by Cascadia Mono.

Banned visual elements

  • Rounded structural corners on cards, callouts, code blocks, tables, page edges. Corners are sharp.
  • Decorative full-width coloured bars at the top or bottom of pages or slides. The system has no banner band.
  • Drop shadows on body content. Shadows do not appear at all.
  • Gradients. The system is flat.
  • Multiple accent hues. Deccan Blue is the only accent. Hierarchy uses opacity.
  • Deccan Green outside the logo or sustainability content (see §04).
  • Stone-tinted page backgrounds in print (see §07).
  • Accent rules under titles in slides (the AI-slop tell). Title hierarchy comes from type, not decoration.

Banned content patterns

See §09. The tone-and-voice ban list is part of the document standard.

11Migration

Documents in flight under earlier conventions migrate to v2.0 on the next regular revision. Three concrete moves:

For documents written under deccan-design v1.0 (Aptos-first)

Replace the font-family declaration with the v2.0 chain. The colour tokens, spatial system, components, and tone rules are unchanged; no body content needs revision. Any explicit Aptos override in inline styles or per-document CSS is removed. The cover and end-page composition is preserved as-is.

For documents written under swiss_design_at_deccan v1.0 (IBM Plex)

Replace the font-family declaration with the v2.0 chain. Remove any inline references to the IBM Plex font binaries, the Hanken Grotesk fallback chain, and the Fira Code mono fallback chain. The colour tokens are identical. Components and spatial rules carry forward. The cover and end-page composition is preserved.

For documents written ad-hoc (no system)

Migrate to v2.0 by adopting the spec wholesale. The transition is from "Word default" or "PowerPoint default" to a deliberate system; the cost is the time to re-style, the value is the next forty documents not relitigating the same decisions.

Template availability

The v2.0 HTML template lives at the same URL as the v1.0 swiss template (github.com/kvkalidindi/swiss_design_at_deccan) and supersedes it. Slot variables are unchanged: {{TITLE}}, {{SUBTITLE}}, {{DOCUMENT_TYPE}}, {{PREPARED_BY}}, {{DATE}}, {{VERSION}}, {{CLASSIFICATION}}, {{BODY_HTML}}. A Word and PowerPoint template emitted from the same tokens will follow.

Adoption test

The system is adopted when the next forty Deccan documents — internal standards, technical specifications, customer-facing briefs, policy memos, audit responses — issue under v2.0 without per-document deviation, and any document that does deviate fails review and is sent back. If that does not happen, the system has not been adopted; it has merely been documented.

12Brand voice

Documents issued under this system speak in Deccan Fine Chemicals' corporate register — direct, declarative, third-person, specific over generic. The register matches the public marketing surface at deccanchemicals.com.

The Deccan register

Deccan describes itself as "a privately owned globally operating company" and the "trusted, transparent and dependable partner" of its customers. The register that produces this is third-person, present-tense, capability-led. The company is the subject; "we" appears sparingly and only on the primary marketing surface.

Two representative sentences from the public site:

Technology and Engineering is our core competency that helps us deliver innovative technological solutions.

Deccan Fine Chemicals makes every product exclusively for a specific customer and does not sell the same product to competitors.

Documents under this system match this register. Sustainability is framed as operational discipline and risk management, not aspiration. Capability is framed as competence — "manufactured by Deccan", "core competency", "delivery" — not aspiration.

Canonical phrasing

The vocabulary below is corporate-canonical and does not need to be re-invented per document:

  • Custom chemical manufacturing — the primary positioning.
  • Exclusive partnership — the customer-engagement model (one product per customer).
  • Active ingredients — manufactured by Deccan, supplied to formulation plants.
  • Formulation plants — the customer-side counterpart to Deccan's intermediates.
  • Privately owned globally operating company — the corporate descriptor.
  • QHSE — Quality, Health, Safety, Environment.
  • AEO-T1 — Authorized Economic Operator – T1 (the customs designation).

Banned patterns

The patterns listed in §09 are not acceptable in any Deccan document. The most common violations — conversational subtitles in headings, first-person narrator asides, filler openings, "deep dive" / "TL;DR", decorative emoji, exclamation marks, mental-exercise framings — appear in machine-generated draft text and are removed at review.

The read-aloud test

If a sentence would feel out of place in a board paper, an audit response, or a regulator submission, rewrite it. The corporate-document register is unambiguous; the test is fast.

13Logo usage

The Deccan Fine Chemicals wordmark identifies every document under this system. Three rules govern its use: clear space, minimum size, and colour variant.

Clear space

The wordmark carries a minimum of 0.5× its own height as clear space on all sides. Within this clear space no other element — type, rule, image, edge — appears. The clear space rule applies in print and on screen.

Minimum size

The wordmark is never rendered smaller than 12 mm wide in print or 80 px wide on screen. Below these thresholds the wordmark loses legibility on standard corporate output devices.

Colour variants

  • Full colour — the default. Deccan Blue rule + Deccan Green sustainability mark + stone-900 wordmark text on a paper background.
  • Monochrome black on paper — single-colour print contexts (faxes, low-end printers, plain-paper photocopies).
  • Monochrome white on Deccan Blue — single-colour print on dark stock; the cover-variant reverse.
  • Sustainability-context — full colour, with the Deccan Green retained per §04.

Banned uses

  • Rotating, skewing, or distorting the wordmark.
  • Recolouring the rule or text to any colour other than the variants above.
  • Placing the wordmark over photographic backgrounds where contrast falls below WCAG AA.
  • Adding drop shadows, glows, bevels, or other ornament.
  • Rendering the wordmark in a face other than Segoe UI Variable / Segoe UI semibold.
  • Cropping or partially obscuring the wordmark.

Assets

Wordmark assets live in skill/assets/:

  • logo.svg — vector wordmark (preferred for print and high-DPI screen).
  • logo.png — 1024×1024 raster fallback.
  • logo.b64.txt — base64-encoded PNG for inline embedding in HTML.

The retrieval cascade is: bundled skill asset → project file → stable raw GitHub URL → base64 fallback. The wordmark is not fetched from deccanchemicals.com; the marketing-site asset is not authoritative for system use.

14File naming and classification

File naming convention

Documents follow:

<owner>-<doc-type>-<topic>-<yyyy-mm>.<ext>

Lowercase. Hyphenated. ISO-style date. Examples:

  • it-policy-acceptable-use-2026-04.docx
  • rd-spec-process-intensification-pilot-2026-05.docx
  • sales-brief-customer-q2-pricing-2026-06.docx
  • qhse-report-emissions-monthly-2026-04.xlsx

This specification is reviewed annually. The owning office (the Office of the SVP, IT & Digital Transformation) re-issues the document with version number incremented on any material change.

  • Quarterly token review — every January, April, July, October. The owner re-validates the colour and type tokens against the marketing site and any updated brand guidelines. Token changes increment the minor version.
  • Ad-hoc revision — issued when a discovered defect, a new OS-native font release, or a corporate decision warrants out-of-cycle change.

Documents issued under this system

Documents written against this specification carry their own version, date, and revision history. They migrate to a new version of the system at their next regular revision per §11.

The adoption test stated in §11 is the canonical measure: the system is adopted when the next forty Deccan documents issue under v2.0 without per-document deviation, and any document that does deviate fails review and is sent back.
